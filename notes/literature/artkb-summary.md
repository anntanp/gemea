## Core Claim

Existing CH knowledge graphs are either institution-specific and static, or general-purpose (Wikidata) without domain-specific structure. No system integrates symbolic RDF knowledge with neural/visual representations in a single operational framework. ArtKB is an end-to-end modular knowledge base that unifies semantic modelling, visual embeddings, and digital asset storage under a unified API gateway, enabling hybrid symbolic+neural retrieval over cultural heritage data.

---

## Architecture

Four components, each addressing a distinct aspect of the CH data lifecycle:

### 1. Graph Database (GraphDB by Ontotext)
- Stores RDF triples expressing semantic relations according to the **CACAO ontology**
- Exposes two query interfaces:
  - **SPARQL 1.1 endpoint** — for expressive semantic queries and relationship traversal
  - **GraphQL endpoint** — for API-oriented consumption by frontend applications
- Enables multi-step relationship queries (artefact → artist → artistic movement → era)

### 2. Object Storage (MinIO)
- Stores binary digital assets: high-resolution images, 3D scans, videos
- Each object assigned a unique identifier linked to its KG entity via `crm:P138_represents`
- Tight coupling: semantic data in GraphDB ↔ binary files in MinIO ↔ vectors in Qdrant
- Provides scalability, data durability, and access management for heterogeneous file types

### 3. Vector Database (Qdrant)
- Stores visual embeddings generated by deep learning models (SwinV2, CLIP)
- Multiple collections per model/strategy for flexibility
- Each vector linked to its KG entity's persistent identifier
- Similarity metric: cosine similarity
- Enables image-to-image, text-to-image, and composed image retrieval

### 4. API Gateway (FastAPI)
- **Single entry point** to all services — no direct inter-service communication
- Manages all external interactions; enforces security/auth
- Coordinates multi-service operations (e.g., ingesting a new artefact: update KG + store file + ingest embedding atomically)
- Endpoints: SPARQL queries, file retrieval from object storage, similarity search via vector DB

## Ontology: CACAO

- **CACAO** (Cultural Artefact Contextual Ontology) — domain-specific extension of CIDOC-CRM
- Introduces explicit subclasses for CH entities (e.g., `cacao:CACAO_0000020` = museum; `cacao:CACAO_0000022` = artist), avoiding CIDOC-CRM's overly generic classes (e.g., E74_Group for museums)
- Integrates **ODRL** (W3C Open Digital Rights Language) for machine-readable rights expressions — critical for legal reuse of digital artefacts
- Entities: physical artefacts, digital artefacts (images), artists, museums, artistic movements, genres, materials
- Aligned with Art & Architecture Thesaurus (AAT) via `skos:exactMatch` for materials

## Knowledge Graph Construction

### Source
Wikidata — filtered to the 7848 entity classes corresponding to CACAO concepts:
physical artefact, artist, museum, movement, genre, material.

### Pipeline
1. Identify Wikidata entity classes mapping to CACAO concepts
2. Filter Wikidata dump to instances of those 7848 base classes
3. Parse each entity type with type-specific property extraction; retain only CACAO-prescribed properties
4. Output: CACAO-aligned RDF triples, reusing Wikidata URIs as identifiers

### Entity modelling highlights

**Artists**: Modelled as `cacao:CACAO_0000022`. Biographical dates modelled as CIDOC-CRM events (`crm:E67_Birth`, `crm:E69_Death`) linked via `crm:P98i_was_born` / `crm:P100i_died_in`. Fields of work, genres, movements retained as separate entities for cross-artist queries.

**Physical artefacts**: Assigned to CACAO subclasses of `cacao:CACAO_0000023`. Creation date and author linked via `crm:E12_Production` event (`crm:P108i_was_produced_by`, `crm:P4_has_time-span`). Materials linked via `crm:P45_consists_of`. Physical dimensions as `crm:E54_Dimension` instances with AAT type links. Depicted subjects via `crm:P62_depicts`; current location via `crm:P53_has_former_or_current_location`.

**Digital artefacts (images)**: Retrieved from Wikimedia Commons; instantiated as `cacao:CACAO_0000086` (subclass of `cacao:CACAO_0000003`). Linked to physical artefact via `crm:P138_represents`. Same identifier used across GraphDB, MinIO, and Qdrant.

**Museums**: Instances of `cacao:CACAO_0000020`. Website via `cacao:CACAO_0000038`. Hierarchical institutional relationships via `crm:P106_is_composed_of`. Curators via `crm:P109_has_current_or_former_curator`.

### Final scale
- **8 million RDF triples**
- **1.4 million unique entities** across **82 classes** (CACAO + CIDOC-CRM + ODRL)
- ~76K artists
- ~307K physical artefacts
- ~2.5K museums
- ~4.8K artistic movements, genres, and materials
- ~147K digital artefacts (images), all linked across GraphDB + MinIO + Qdrant

## Use Cases

### 5.1 Semantic Knowledge Discovery (Text2SPARQL)
**Problem**: Non-technical CH users can't write SPARQL but have precise structured queries in mind.

**Implementation**: LLM-based Text2SPARQL with 2-stage entity resolution:
1. **LLM** (Mistral Small, temperature 0.1): receives ontology structure + task description → translates NL query to SPARQL with entity placeholders (e.g., artist name becomes a placeholder, not a hard-coded IRI)
2. **Entity resolution**: SPARQL label lookups resolve placeholders to concrete IRIs (e.g., "Vincent Van Gogh" → `wd:Q5582`)
3. Final SPARQL query executed against GraphDB

**Effectiveness depends on** metadata completeness — comprehensive temporal, material, provenance fields improve precision.

### 5.2 Neural-based Image Retrieval
Three retrieval modes:
- **Image-to-image**: find visually similar artefacts to a query image (SwinV2 embeddings → Qdrant k-NN)
- **Text-to-image**: find images matching a textual description (CLIP embeddings → Qdrant)
- **Composed Image Retrieval (CIR)**: query = reference image + textual modification instruction (MagicLens, LamRA); outputs modified retrieval results

### 5.3 Neural-aided Metadata Suggestion
**Problem**: Digitising CH artefacts requires manually entering many metadata fields; time-consuming for curators.

**Process**:
1. Input image → SwinV2 → vector embedding
2. k-NN search in Qdrant → top-k most visually similar artefacts
3. Retrieve metadata of those artefacts from GraphDB
4. **Similarity-weighted majority voting** aggregates metadata values → ranked suggestions per field (artist, material, type, movement)
5. Curator selects from suggestions

**Applicable to** any categorical attribute the curator wants to fill.

### 5.4 Text-to-Graph Generation (Text2RDF)
**Problem**: Unstructured textual descriptions of artefacts aren't in the KG.

**Implementation**: Multi-agent LLM pipeline (Google Agent Development Kit):
1. **Specialised LLM agents** prompted with summarised CACAO ontology structure perform:
   - Named entity recognition (NER)
   - Coreference resolution
   - Relation extraction
2. **Entity linking**: SPARQL lookups against authoritative vocabularies (Wikidata, AAT) and internal KG → top-N candidate IRIs per entity
3. **RDF triple generation**: extracted entities/relations → CACAO-compliant RDF triples
4. **Critic agent**: self-reflective validity check against CACAO before merging into the KG

## Results / Evaluation

The paper presents qualitative demonstrations rather than quantitative benchmarks. Use case effectiveness is illustrated with examples (e.g., Titian's Venus of Urbino for text-to-image, Cupid and Psyche sculpture for metadata suggestion). No precision/recall numbers reported for Text2SPARQL or retrieval tasks — identified as future work.

---

## Limitations

- **Scale**: 307K physical artefacts (single Wikidata source, art domain only)
- **No quantitative evaluation**: retrieval and Text2SPARQL quality not benchmarked with metrics
- **No web UI**: API-only; no described end-user browsing interface
- **Single source**: no cross-institutional heterogeneity; all data normalized from Wikidata
- **No named graphs**: flat graph, no provenance partitioning by source/batch
- **No self-hosting documentation**: MIT code but no Docker Compose or deployment guide
- **GraphDB is commercial**: limits fully open deployment for community users
- **Static corpus**: no described mechanism for incremental updates from Wikidata changes
- **ODRL integration**: mentioned but not deeply evaluated for correctness or coverage

## Future Work (stated)

- Expand coverage and depth of the KG (more entities, richer properties)
- Develop and evaluate multimodal retrieval models further
- Align with the **European Collaborative Cloud for Cultural Heritage (ECCCH)**
- Evaluation with institutional partners for usability and scalability
- Deployment as a robust, open, intelligent platform for CH data management
